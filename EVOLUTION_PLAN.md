## 🎯 设计哲学

### 核心原则

1. **长期主义** - 不追求短期功能堆砌，注重架构的可持续性
2. **代码质量** - 测试覆盖、代码规范、文档完善是硬性要求
3. **稳步演进** - 小步快跑，每个里程碑都有明确的交付物
4. **用户价值** - 每个功能都要解决真实问题，不为技术而技术
5. **简单优先** - 保持代码简洁，避免过度设计（Linus原则）

### 质量标准

- **测试覆盖**: 核心功能 ≥ 90%，整体 ≥ 80%
- **代码规范**: Ruff 检查零警告
- **文档完善**: 每个新功能都有使用文档和开发文档
- **向后兼容**: 公共API保持向后兼容，重大变更需经过 deprecation 周期

## 📊 质量保障体系

### 测试策略

- **单元测试**: 核心逻辑必须有单元测试
- **集成测试**: 适配器层需要集成测试
- **E2E 测试**: WebUI 关键流程需要 E2E 测试
- **性能测试**: 定期进行性能基准测试

### 代码审查

- 每个功能分支都需要 PR
- PR 必须通过所有测试
- 至少一个人审查（虽然现在是我一个人，但要保持这个习惯）

## 📝 开发工作流

### 每个功能的开发步骤

1. **需求分析** - 明确要解决什么问题
2. **设计讨论** - 考虑架构影响和实现方案
3. **测试先行** - 先写测试用例
4. **实现功能** - 编写生产代码
5. **文档更新** - 更新使用文档和开发文档
6. **代码审查** - 自我审查和改进
7. **合并发布** - 合并到主分支并发布
